# 特征工程与筛选方法论

## 为什么要调整

如果只挑少量看起来有业务含义的变量，再用模型结果证明它们有效，会有“先射箭再画靶”的风险。真实风控项目里，变量通常不是一开始就顺利选出来的，而是先从可用数据源系统生成候选变量，再通过稳定性、区分度、相关性、泄露风险和业务可解释性逐层筛选。

本项目因此采用两阶段路线：

1. **候选特征工厂**：尽量系统地产生申请主表、外部征信、历史申请、还款纪律、信用卡和 POS/cash loan 的候选变量。
2. **客观筛选漏斗**：先用预先定义的统计规则筛选，再进入模型重要性和业务 reason code 解释。

## 借鉴高分方案的部分

- 不只使用 `application_train/test`，而是充分使用全部历史表。
- 不预设少数“好看”的变量，而是生成大量聚合、比率、计数、状态、缺失和类别编码特征。
- 先聚合到 `SK_ID_CURR` 申请粒度，再统一筛选。
- 对类别变量进行编码，而不是只保留原始字符串。
- 通过稳定性、缺失率、单变量区分度和相关性去重压缩特征空间。

## 本项目的筛选漏斗

### Stage 0: 防泄露和可用性

- `TARGET` 不进入特征。
- `SK_ID_*` 只用于连接，不作为模型特征。
- 历史表特征初步视为申请时点可用，但后续继续复核时间窗口。
- 对可能是贷后才产生的字段设置人工复核标记。

### Stage 1: Target-blind 质量筛选

不看标签，只看数据质量：

- 缺失率过高的变量降级或剔除。
- 训练集无变化或近似常数变量剔除。
- 在 `application_train` 内部切分 development / validation / final_holdout，并用 development/validation PSI 检查开发期稳定性。
- `final_holdout` 不参与筛选，留给模型完成后的最终 PSI 诊断和“移除漂移变量前后”的表现对比。
- 官方 Kaggle test 是无标签外部样本，不参与任何特征选择、模型选择或 watchlist 生成。
- 类别变量先做频率编码和 top 类别 one-hot。

### Stage 2: 单变量区分度筛选

只在训练集上看标签：

- AUC power：`max(AUC, 1 - AUC)`。
- KS：好坏样本累计分布差异。
- IV：分箱后的证据权重强度。
- 当前采用宽松筛选口径，保留弱单变量但有业务含义的近期窗口和分组聚合变量，交给 LightGBM 继续判断交互价值。

这些指标只用于形成候选短名单，不直接作为最终业务结论。

### Stage 3: 冗余特征去重

- 对通过初筛的数值特征做相关性检查。
- 高相关变量中优先保留区分度更高、缺失更低、业务解释更清楚的变量。
- 保存高相关变量对和去重决策表，确保每个剔除动作可追溯。

### Stage 4: 模型筛选

下一轮使用 LightGBM / CatBoost 等树模型：

- 交叉验证特征重要性。
- permutation importance 或 null importance。
- 删除不稳定或贡献弱的特征。
- 模型冻结后，在 `final_holdout` 上检测 PSI，并对高 PSI 变量做移除前后的模型表现 ablation。

### Stage 5: 业务解释与策略化

只有通过前面筛选的变量，才进入业务解释层：

- 变量是否能形成 reason code。
- 是否能支持 approve / manual review / decline。
- 是否能被监控，是否容易产生政策歧视或不可解释风险。

## 面试表达

可以这样说：

> 我没有直接挑几个看起来合理的变量建模，而是把特征工程拆成候选变量工厂和筛选漏斗。先系统生成多表聚合、比率、状态、缺失和类别编码变量，再用缺失率、稳定性、AUC/KS/IV、相关性去重和后续模型重要性筛选，最后才把留下来的变量转成风控 reason code。这样可以避免先有业务故事再倒推变量。
